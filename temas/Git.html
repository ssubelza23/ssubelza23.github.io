Git

Git es un sistema de control de versiones distribuido de código abierto creado en 2005 por Linus Torvalds y otros de la comunidad de desarrollo de Linux. Git puede funcionar con muchos tipos de proyectos, pero se usa más comúnmente para el código fuente del software.

Introduccion

El control de versiones es un sistema que realiza un seguimiento de los cambios en un archivo o grupo de archivos a lo largo del tiempo. Cuando tiene un historial de estos cambios, le permite encontrar versiones específicas más adelante, comparar cambios entre versiones, recuperar archivos que pueda haber eliminado o revertir archivos a versiones anteriores. Un sistema de control de versiones distribuido significa que diferentes usuarios mantienen sus propios repositorios de un proyecto, en lugar de trabajar desde un repositorio central. Los usuarios tienen automáticamente capacidades completas de seguimiento de archivos y el historial completo de versiones del proyecto sin necesidad de acceder a un servidor central o red. Cuando Git se inicializa en un directorio de proyecto, comienza a rastrear los cambios del archivo y los almacena como "conjuntos de cambios" o "parches". Los usuarios que trabajan juntos en un proyecto envían sus conjuntos de cambios que luego se incluyen (o rechazan) en el proyecto. freeCodeCamp tiene una lista de reproducción de YouTube con muchos consejos prácticos de Git y GitHub aquí .

Tipos de Sistemas de control de versiones

Sistemas de Control de Versiones Cetralizado

Los sistemas centralizados tienen una copia del proyecto alojado en un servidor centralizado, al que todos se conectan para realizar cambios. Aquí, se adopta el principio de "primero llegado, primero servido": si usted es el primero en enviar un cambio a un archivo, su código será aceptado.

Sistemas de Control de Versiones Distribuido

En un sistema distribuido , cada desarrollador tiene una copia de todo el proyecto. Los desarrolladores pueden realizar cambios en su copia del proyecto sin conectarse a ningún servidor centralizado y sin afectar las copias de otros desarrolladores. Más tarde, los cambios se pueden sincronizar entre las distintas copias.

Ventajas de los sistemas de control de versiones Distribuidos

Torvalds insistió en un sistema distribuido debido a la independencia que brinda a los desarrolladores. Con un sistema distribuido, puede trabajar en su copia del código sin tener que preocuparse por el trabajo en curso en el mismo código por parte de otros. Lo que lo hace aún mejor es que cualquier copia distribuida del proyecto puede contener todo el historial del proyecto. Un sistema distribuido también le permite trabajar sin conexión, lo que significa que puede realizar cambios sin tener acceso al servidor que almacena el repositorio central.

Otra ventaja de los sistemas distribuidos es que pueden sincronizar sus repositorios entre ustedes, sin pasar por la ubicación central. Digamos que el acceso al servidor principal se cae y tienes que colaborar con un colega. Puede compartir los cambios con su colega y continuar trabajando juntos en el proyecto, y luego enviar todos los cambios a la ubicación a la que todos tienen acceso.

En un sistema centralizado, cualquier persona que realice un cambio debe tener acceso a la ubicación central. Por el contrario, en un sistema distribuido, los nuevos desarrolladores pueden realizar cambios en sus propios repositorios sin tener acceso de escritura, mientras que los contribuyentes más experimentados pueden tener acceso de escritura y la capacidad de revisar otras contribuciones antes de fusionarlas en el repositorio. Administrar el acceso es más fácil en sistemas distribuidos.

Git y Github

La gente a menudo confunde a Git y GitHub, pero son cosas muy diferentes. GitHub proporciona servicios relacionados con Git. Es un sitio web que lo ayuda a administrar proyectos controlados por Git.

GitHub permite a los usuarios colocar sus repositorios de Git en la nube y realizar operaciones basadas en Git a través de una interfaz web. También proporciona aplicaciones de escritorio y móviles que ofrecen los mismos servicios. GitHub se lanzó unos años después de Git, y sigue siendo muy popular entre los entusiastas del código abierto.

Empezando con Git

Instalacion

El primer paso es instalar Git. El sitio web oficial de Git proporciona instrucciones detalladas sobre cómo instalar Git en su máquina local, dependiendo de su sistema operativo.

  • Si está utilizando Linux, puede instalar Git a través del terminal utilizando un administrador de paquetes. Para la popular distribución de Linux Ubuntu, Git se puede instalar usando : apt-get

    apt-get install git

  • En OS X, si tiene Homebrew , puede instalar Git usando la línea de comando a través del siguiente comando:

    brew install git

  • Si está en Windows, la versión oficial de Git se puede descargar desde el sitio web de Git.

El Flujo de Trabajo de GIt

Hay tres operaciones básicas realizadas por Git en su proyecto:track, stage, y commit.

  • Track: Una vez que haya inicializado su repositorio, deberá agregar archivos a su proyecto

  • Stage: El Staging es un estado de preparación antes de hacer commit. Aquí guardamos las actualizaciones y nuevas incorporaciones de archivos al proyecto con el comando git add. Son archivos en el área de Staging. Viven dentro de Git, tienen historial de cambios y sus últimas actualizaciones están listas para guardarse definitivamente en un nuevo commit cuando hagamos git commit.

  • Commit: El siguiente paso es crear un commit. Una commit es como una fotografía que registra el estado actual de su código.

Primeros comandos

Establecer Ajustes de Configuracion

Antes de continuar con el uso de Git en un proyecto, definamos algunas configuraciones globales:

git config --global user.name "mamba" git config --global user.email "manba@gmail.com" git config --global

Si desea verificar todas las configuraciones para su repositorio, puede ejecutar el siguiente comando:

git config --list
